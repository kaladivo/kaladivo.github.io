Linky top-up bug - a plain-English explanation

You paid the Lightning invoice. Nothing showed up in Linky. Here is exactly why.

A Cashu top-up is not one action - it is three separate calls between your browser and the mint. For about six weeks, Linky's production build could complete only the first one. For the final five of those weeks, the default mint was one that made the breakage hit almost every new top-up.

Symptom

Invoice paid, no sats arrive

The Receive screen shows a Lightning invoice, your wallet pays it, and then the Linky balance never updates.

Cause

Two of three mint calls were blocked

Invoice creation was routed through Linky's own server, but the two follow-up calls that actually mint ecash were still sent straight from the browser to the mint - and the browser refused them.

Trigger

Default mint changed

The previous default mint happened to be permissive, so the gap stayed hidden. When the default moved to cashu.cz, the gap became the normal experience.

What actually happens during a Cashu top-up

A top-up is a three-step conversation between your browser and the mint.

Paying the Lightning invoice is the middle of the story, not the end of it. Before paying, the app has to ask the mint for a quote. After paying, the app has to go back to the mint twice more: once to confirm the payment was received, and once to actually mint the ecash tokens into your wallet. Until that last call completes, no Cashu proofs exist yet. The mint is just holding your sats, waiting to issue tokens.

Your browser (Linky)
Call
The mint
1. "I want to deposit X sats." The app asks for an invoice to top up the wallet.
POST
/v1/mint/quote/bolt11
Returns a Lightning invoice + quote ID. The quote ID is how the app will refer to this deposit later.
You pay the invoice with any Lightning wallet. The mint receives the sats. Linky is not involved in this step - this is pure Lightning.
2. "Is quote X paid yet?" The app polls until the mint says the invoice has been settled.
GET
/v1/mint/quote/{id}
Returns state: PENDING -> PAID. Once PAID, the mint is ready to issue Cashu tokens against it.
3. "Please mint my tokens now." The app sends blinded requests for ecash denominations.
POST
/v1/mint
Returns signed Cashu proofs. The app stores these locally. This is when the balance actually appears.

Key point: paying the Lightning invoice does not put Cashu in your wallet. Paying only tells the mint it has your sats. Steps 2 and 3 are where the actual ecash is created. Skip them and the balance stays at zero forever.

The hidden rule that broke the flow

Browsers will not let a web app call any server it wants.

This is the whole secret of the bug. When a webpage in your browser tries to call a server on a different domain, the browser applies a security check called CORS (Cross-Origin Resource Sharing). The target server has to explicitly say "yes, I accept calls from this origin" via HTTP headers. If it does not, the browser quietly refuses the call before it even happens.

Analogy

CORS is your browser being paranoid on your behalf. Imagine a bank that only accepts instructions sent through its own verified phone lines. If a random third-party caller rings in on an unverified number, the bank refuses to even listen. Cashu mints are the same: unless the mint explicitly welcomes browser calls from app.linky.fit, your browser treats them like that unverified caller and hangs up.

Most Cashu mints have never configured CORS, because they assume clients are native apps, server-to-server integrations, or scripts - environments where CORS simply does not apply. Browsers are the odd one out.

The workaround: a proxy on your own domain

A proxy here is a tiny server that runs on Linky's own domain. The browser calls the proxy instead of the mint. Since the proxy lives on the same origin as the app, CORS does not apply. The proxy then forwards the request to the mint from server to server, where CORS also does not apply. The mint answers, the proxy passes the answer back, and the app gets what it needed.

Before the bug existed

Linky's first top-up never called the mint from the browser at all.

In the original design, the Receive screen used the user's npub.cash Lightning address. The app asked npub.cash for a Lightning invoice, showed it, and when paid, npub.cash delivered the ecash back to the app through an existing sync mechanism. The mint was in the picture, but only through npub.cash - not through the browser.

Browser(Linky app)

npub.cash(LN address service)
->
Mint(behind the scenes)

CORS was never on the critical path for top-up. That is why early Linky could work in a browser even though a direct-to-mint call would have been refused.

Commit 474fd0e - 2026-01-16

We switched Linky to talking directly to the mint.

A redesign introduced a proper Cashu-native receive flow. The app started performing all three mint calls itself. This was the correct Cashu protocol flow, and it enabled big wins: users could pick any mint they wanted, restore deterministic wallets, control denominations, and so on.

The tradeoff: the browser now had to reach the mint directly, so CORS became a hard dependency. If the chosen mint did not welcome browser calls, top-up would fail. A developer added a Vite dev proxy a week later (52b3b88) so local development worked, but nothing equivalent existed in production.

Call 1

Create quote

POST /v1/mint/quote/bolt11 - direct from the browser to the mint.

Call 2

Check quote

GET /v1/mint/quote/{id} - direct from the browser.

Call 3

Mint proofs

POST /v1/mint - direct from the browser.

At this point the bug was technically present for any mint that did not allow browser calls. It stayed invisible because Linky's default mint (mint.minibits.cash) happened to send permissive CORS headers and therefore accepted browser calls anyway.

Commit b81c4a3 - 2026-03-15

We added a proxy - but only for the first call.

A new serverless endpoint, /api/mint-quote, was deployed alongside the app. The Receive screen started calling Linky's own server for invoice creation instead of calling the mint directly. That fixed step 1 for every mint, CORS or not.

But the two follow-up calls that actually mint the ecash - wallet.checkMintQuote() and wallet.mintProofs() - were untouched. The Cashu library (cashu-ts) still sent those straight from the browser to the mint using its default request handler.

Fixed in this commit

Call 1 - Create quote

/api/mint-quote?mint=...
  • Browser sends the request to Linky's own origin.
  • The serverless route forwards it to /v1/mint/quote/bolt11.
  • CORS no longer blocks invoice creation.
Still broken after this commit

Calls 2 and 3 - Check + Mint

wallet.checkMintQuote + wallet.mintProofs
  • The wallet library still calls the mint directly.
  • The browser can still block both requests.
  • Even a paid invoice never gets claimed as ecash.
# Before b81c4a3 (production) Browser -> mint /v1/mint/quote/bolt11 (needs CORS) Browser -> mint /v1/mint/quote/{id} (needs CORS) Browser -> mint /v1/mint (needs CORS) # After b81c4a3 (production) Browser -> /api/mint-quote -> mint (fixed) Browser -> mint /v1/mint/quote/{id} (still direct) Browser -> mint /v1/mint (still direct)

Why this slipped through: if you manually tested by generating an invoice and paying it, the invoice definitely appeared. And if you were on a mint with permissive CORS (like minibits), even the follow-up calls silently worked. The commit looked complete.

Commit fb1fb98 - 2026-03-20

Then the default mint changed to cashu.cz - and the gap became everyone's problem.

A one-line commit swapped the default mint URL. Small change, huge effect. The old default (mint.minibits.cash) had permissive CORS headers, which quietly masked calls 2 and 3 still being direct. The new default (cashu.cz) does not send those headers. From this point on, the most common receive path hit a wall on step 2.

- export const MAIN_MINT_URL = "https://mint.minibits.cash/Bitcoin"; + export const MAIN_MINT_URL = "https://cashu.cz";
Before this commit

Bug exists, but hidden

Only users who had manually chosen a strict mint would see the failure. Most users defaulted to minibits, and minibits accepted the browser's direct calls, so top-up looked fine.

After this commit

Bug hits the default path

New users, factory-reset users, and anyone opted into the default mint now topped up through a mint whose browser calls were refused. Invoice appeared, invoice was paid, nothing landed.

Why the bug felt so confusing

Two parallel timelines: what you saw, and what the app was trying to do.

From the user's perspective, every sign said "this worked" - the invoice appeared, the Lightning payment went through, the payer's wallet showed a successful send. And then: nothing. The app looked broken in the most frustrating way, because the failure was silent. Here is the two-track view.

What the user saw
  • Invoice appeared Entered amount, QR and string showed up as expected.
  • Lightning wallet paid it Payer's wallet reported success.
  • Nothing happens Balance stays at previous value. No error. No spinner forever.
  • Close and reopen - still nothing Reopening the app does not help, because the same CORS-blocked calls run again and fail the same way.
What the app was doing
  • Call 1: quote created Went through /api/mint-quote. Worked.
  • (Lightning delivered sats to the mint) This happens outside the app. Nothing to fail here.
  • Call 2: check quote - blocked by CORS Browser refused to contact cashu.cz directly. The app never found out the invoice was paid.
  • Call 3: mint proofs - never ran Because call 2 never succeeded, the app had no "PAID" signal to trigger minting. No proofs were created.

Good news: the sats are not lost. The mint is holding them against the paid quote. As soon as Linky can run calls 2 and 3 from a context where CORS does not block them - for example after the fix below - the ecash shows up in the wallet.

Commit 544bf9f - 2026-04-24

The real fix: route every mint call through Linky's own server.

The fix does what the half-fix should have done six weeks earlier. A second serverless route, /api/cashu-mint, acts as a generic proxy for any mint endpoint. Instead of hardcoding a single path (like /api/mint-quote did), this one takes ?mint= and ?path= query parameters and forwards the request accordingly.

On top of that, the Cashu wallet bootstrap now accepts a useWebProxy: true flag. When the web app constructs the wallet, it installs a custom request handler into cashu-ts that rewrites every mint call - including the two that were broken - to go through /api/cashu-mint. Native shells (Android, iOS) do not set this flag, because they are not browsers and do not have CORS restrictions.

Call 1

Create quote

Browser -> /api/cashu-mint -> mint.

Call 2

Check quote

Browser -> /api/cashu-mint -> mint.

Call 3

Mint proofs

Browser -> /api/cashu-mint -> mint.

# After 544bf9f (production, web) Browser -> /api/mint-quote -> mint (existing) Browser -> /api/cashu-mint -> mint (new, covers check + mint) # Native apps (Android, iOS) - unchanged Native -> mint directly (no browser, no CORS)

With all three calls proxied, the flow finally works end to end on the default mint. Any previously paid-but-unclaimed quote is picked up on next launch and converted into spendable ecash.
