Linky receive bug

Invoice creation was fixed, but invoice claiming stayed broken.

The old receive flow worked because the browser did not need to talk directly to a Cashu mint. Later, Linky switched to mint-native top-up quotes. Commit b81c4a3 proxied only the first request, and commit fb1fb98 made cashu.cz the default mint.

Working path

Original receive did not depend on browser-to-mint CORS.

The early top-up feature used the user's `npub.cash` Lightning address. Linky asked the LNURL service for an invoice, the payer paid it, and later Linky recovered Cashu value through that service.

Browser app
Set amount User opens Receive and chooses sats.
Ask npub.cash `fetchLnurlInvoiceForLightningAddress` gets invoice.
Show invoice User pays a BOLT11 invoice.
Mint CORS
Not required The browser is not calling mint quote or mint proof endpoints here.

This is why it could work before even if direct mint calls from a production browser would have failed.

Design change

The custom mint receive flow moved the app onto mint-native quotes.

Commit 474fd0e introduced the direct mint flow: create a mint quote, show the returned invoice, poll the quote, then mint proofs after payment. That path is correct for Cashu, but in a browser every direct cross-origin mint request needs the mint to allow CORS.

Browser app
Create quote `POST /v1/mint/quote/bolt11`
Check quote `wallet.checkMintQuote(quoteId)`
Mint proofs `wallet.mintProofs(amount, quoteId)`
Requirement
CORS now matters If the mint does not allow the app origin, browser fetch fails.
Commit b81c4a3 - 2026-03-15

`/api/mint-quote` fixed only the first mint request.

This commit added a serverless API route and changed quote creation to always call the app origin. That means the browser calls Linky, and Linky's server calls the mint.

Fixed

Quote creation

/api/mint-quote?mint=...
  • Browser sends amount to Linky's own origin.
  • Server route forwards to `/v1/mint/quote/bolt11`.
  • Browser CORS no longer blocks invoice creation.
Not fixed

Paid quote claim

checkMintQuote + mintProofs
  • Cashu wallet still calls the mint from the browser.
  • The paid invoice can exist, but polling can fail.
  • Minting proofs can also fail before sats are saved.
Before b81c4a3: DEV browser -> /__mint-quote -> mint PROD browser -> mint /v1/mint/quote/bolt11 After b81c4a3: PROD browser -> /api/mint-quote -> mint /v1/mint/quote/bolt11 PROD browser -> mint check quote (still direct) PROD browser -> mint proofs (still direct)
Commit fb1fb98 - 2026-03-20

The default mint changed to cashu.cz.

This commit is small but important. It changed one constant: `MAIN_MINT_URL` moved from Minibits to `https://cashu.cz`.

- export const MAIN_MINT_URL = "https://mint.minibits.cash/Bitcoin"; + export const MAIN_MINT_URL = "https://cashu.cz";
Before

Only some users hit cashu.cz

Users had to select or already have cashu.cz as their receive mint.

After

Default receive path hits cashu.cz

New/default receive attempts now depend on cashu.cz allowing all required browser calls.

Failure shape

Why the bug looked confusing.

A paid invoice does not mean the app has minted and stored Cashu proofs. It only means the Lightning invoice was paid. The wallet must still check the quote state and mint proofs.

What user sees
Invoice appears `/api/mint-quote` made this work.
Invoice paid Lightning payment reaches the mint quote.
No sats appear Proof minting never completes in the browser.
Hidden app work
Create quote Proxied through Linky.
Check paid Still direct browser-to-mint.
Mint proofs Still direct browser-to-mint.

So the feature did not fail at "make an invoice". It failed after payment, at "turn the paid quote into local Cashu proofs".
