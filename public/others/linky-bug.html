The receive bug sits at the boundary between Linky's app logic, mint CORS behavior, and Cashu's bearer-token security model. We can prove from git that Linky half-proxied quote creation in b81c4a3. Verified

cashu.cz supports browser CORS now

With an Origin: https://app.linky.fit request header, the relevant /v1/ endpoints return Access-Control-Allow-Origin: *.

Important

Plain curl can mislead

A request without an Origin header may not show CORS headers. That is normal. Browsers send Origin on cross-origin fetches.

Code bug

Quote status was cacheable

The app checked UNPAID -> PAID with a normal GET. Without cache: "no-store", a browser can keep returning an old UNPAID response.

What we know now

The current cashu.cz CORS behavior is permissive for Linky-style browser requests.

The live checks were made with the same kind of header a browser sends for a cross-origin request: Origin: https://app.linky.fit. Those checks returned CORS headers for mint info, keys, keysets, quote creation, quote check, and minting preflight.

# Request shape that matters curl https://cashu.cz/v1/info \ -H "Origin: https://app.linky.fit" access-control-allow-origin: * access-control-expose-headers: *

If cashu.cz was failing before, the likely cause was its server-side CORS middleware/config at that time, a transient deployment difference, or a misleading test that did not include an Origin header. Cashu itself does not guarantee CORS; each mint has to configure it.

Receive mechanics

A top-up is not complete when the Lightning invoice is paid.

Paying the invoice gives the mint sats. Linky still has to claim Cashu proofs from the mint. The app performs three mint operations: create the quote, check the paid quote, then mint proofs.

Call 1

Create quote

POST /v1/mint/quote/bolt11

Call 2

Check quote

GET /v1/mint/quote/bolt11/{quote}

Call 3

Mint proofs

POST /v1/mint/bolt11

This is why the symptom can be "invoice paid, no sats in Linky": Lightning succeeded, but the browser still needed to complete the Cashu claim.

Before the direct mint path

The earliest top-up path avoided browser-to-mint CORS entirely.

Early Linky top-up used the user's npub.cash Lightning address. The browser talked to an LNURL service, not directly to the mint's Cashu endpoints. That is why it could work in browsers even if a direct Cashu mint call would have been blocked.

The older HTML presentation said this proved cashu.cz did not send CORS headers. That was too strong. The commit proves only the default mint changed. The CORS state is a runtime property of the mint server.

Why CORS appeared inconsistent

CORS was a plausible suspect, but caching is the concrete app bug.

Servers often add CORS headers only when the incoming request has an Origin header. That means a plain server-to-server request can show no CORS headers, while the same endpoint responds correctly to a browser-style request.

Misleading test

No Origin header

curl https://cashu.cz/v1/info may show no Access-Control-Allow-Origin. That does not prove browsers are blocked.

Relevant test

Browser-style Origin

curl ... -H "Origin: https://app.linky.fit" currently returns permissive CORS from cashu.cz.

If the Android browser failed earlier, the strongest app-side explanation is stale quote-status caching. The cashu.cz status endpoint currently returns CORS headers, but it does not return Cache-Control: no-store. Linky must force no-store on the fetch that polls whether the quote has become paid.

The fix we want now

Use direct CORS fetches with no-store. Do not route Cashu secrets through Linky.

A generic mint proxy would hide CORS problems, but it introduces a worse trust problem. Cashu proofs are bearer assets. Mint quote IDs are also sensitive during top-up. If a proxy sees those values, it can theoretically claim or steal value. The targeted fix is to make every mint call direct and explicitly uncached.

BrowserLinky app
->
Cashu mintdirect CORS request
# Current fix direction fetch(mintEndpoint, { mode: "cors", credentials: "omit", cache: "no-store", headers: { Accept: "application/json", "Content-Type": "application/json" } }) # Removed from Linky /api/mint-quote /api/cashu-mint

This matches the current live behavior of cashu.cz, avoids exposing quote IDs or proofs to Linky servers, and still logs likely CORS/network failures if a selected mint is not browser-compatible.
