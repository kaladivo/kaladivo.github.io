Gradual web app rewrite

Make Linky faster to change without stopping product delivery

The goal is not a long rewrite project. The goal is to move business logic into a clean core package, replace risky parts of the current web app piece by piece, and keep shipping while the architecture improves.

Target path

1
Current web app Feature-rich, but hard to understand and debug.
2
Current web app using clean core Lower risk, better tests, clearer ownership.
3
New web app using clean core Modern UI code on top of proven business logic.

Business problem

The current cost is not just technical debt

When core behavior is tangled with UI, storage, and platform code, every product change takes longer and carries more risk.

Delivery risk

Small changes are expensive

Developers need to understand too much unrelated code before safely changing one product flow.

Quality risk

Bugs are harder to isolate

Payment, messaging, identity, and UI behavior currently overlap, making regressions harder to detect early.

Planning risk

Estimates become unreliable

Product work is harder to size when feature boundaries are not reflected in the code.

Business framing

This rewrite is an investment in delivery speed, product quality, and future platform flexibility, not a cosmetic code cleanup.

Recommended approach

Rewrite the business logic first, then the app shell

A big-bang rewrite would pause value delivery and increase launch risk. The safer route is to extract stable product logic into @linky/core and let the existing web app consume it incrementally.

Stabilize

Capture current behavior in focused tests before changing the implementation.

Extract

Move product rules into core with clear domain boundaries and adapter interfaces.

Replace

Wire one feature flow at a time back into the existing web app.

✓
No months-long rewrite freeze Product work continues while architecture improves.
✓
Core owns business decisions React, browser APIs, storage, and device bridges stay in the app layer.
✓
Each step is releasable Every chunk should leave the product working and tested.

Operating model

Separate product rules from delivery channels

The core package becomes the source of truth. The web app becomes one consumer of that logic, alongside possible future native, desktop, or service-side consumers.

Core package

What core owns

Identity derivation, contacts rules, messaging protocol, wallet/token decisions, payment orchestration, validation, retention, and domain tests.

Web app

What the app owns

React screens, routing, Evolu adapters, browser storage, Capacitor bridges, service worker integration, translation, and presentation.

Why this matters

Once business logic is portable and tested, future UI redesigns or native product work become much smaller projects.

Replaceable chunks

Break the rewrite into product domains

These chunks are small enough to migrate gradually, but meaningful enough to give product and engineering clear ownership.

1

Identity and owner lanes

Account creation, seed recovery, Nostr keys, Cashu seed, and local-first owner routing.

2

Contacts

Contact rules, limits, unknown sender handling, blocking, and profile identifiers.

3

Messaging protocol

Nostr message creation, parsing, replies, edits, reactions, and malformed-message rejection.

4

Messaging storage and sync

Inbox sync, pending sends, dedupe, retention, reactions, and unknown-thread migration.

5

Wallet inventory

Cashu token states, spendable balance, reserved/externalized tokens, and mint grouping.

6

Cashu operations

Accept, check, restore, send, melt, top-up claim, counters, and recovery paths.

7

Lightning and LNURL

Lightning address, invoices, withdraw flows, amount limits, and payment capability checks.

8

Payments orchestration

Pay contact, pay unknown recipient, payment requests, decline, publish, and result recording.

9

Notifications and telemetry

Payment notification classification and anonymous event construction without blocking product flows.

10

App shell and routing

Replace the large composition layer only after core behavior is proven and reusable.

Recommended sequence

Start with low-risk, high-leverage domains

The first phases should improve confidence and reduce complexity before touching the highest-risk money-moving flows.

Phase 1

Identity and owner lanes

Finish the domain already started in core. This creates a clear template for future extractions.

Phase 2

Contacts and messaging protocol

Move pure rules and protocol behavior first. This improves test coverage without changing payment execution.

Phase 3

Messaging storage and sync

Replace the complex message domain after the protocol rules are stable and independently tested.

Phase 4

Wallet inventory, Cashu, and Lightning

Move money-related behavior in narrow slices with strong regression tests around current product behavior.

Phase 5

New app shell

Once core is stable, rebuild the web app around clean feature modules instead of the current large composition layer.

Execution playbook

Every chunk follows the same migration loop

A repeatable process makes progress visible and keeps each change releasable.

01

Capture behavior

Add characterization tests for the current flow so existing product behavior is protected.

02

Move rules to core

Extract domain types, validation, decisions, and use cases into the core package.

03

Add app adapters

Keep Evolu, browser storage, relays, and native bridges in the web app behind clear interfaces.

04

Replace one flow

Wire the existing UI to the new core use case and remove duplicate old logic after validation.

Success measures

How we know the rewrite is working

Success should be visible in delivery, quality, and planning, not only in cleaner folders.

Delivery

Faster feature changes

Product changes touch fewer files and require less unrelated context.

Quality

More behavior under tests

Critical payment, messaging, and identity rules can be tested without running the whole web app.

Product

Lower release anxiety

The team can change one domain without accidentally rewriting unrelated product behavior.

Architecture

Clear feature ownership

Code lives near the feature it supports, making onboarding and debugging easier.

Strategy

Reusable core

Future web, native, or service-side product surfaces can share the same tested logic.

Risk

No rewrite cliff

Progress ships incrementally instead of waiting for a single high-risk replacement launch.

Risks and controls

The main risk is migration discipline

Gradual rewrites succeed when the team avoids parallel systems that never converge. Each chunk needs a clear finish line.

Risk

Core becomes another dumping ground

Control: core modules must be domain-based, tested, and free of React, browser, and storage-specific code.

Risk

Old and new logic both remain forever

Control: every migration ticket includes deletion of the replaced code path after validation.

Risk

Money flows regress

Control: payment and wallet chunks move later, in small slices, with behavior tests before implementation changes.

Risk

Scope expands into a full redesign

Control: the app shell is last. Core extraction comes first, and user-facing redesign work stays separate unless explicitly planned.

Decision request

Approve a gradual rewrite as a product enablement track

The proposal is to fund the rewrite as ongoing incremental work, not as a separate months-long project that blocks roadmap delivery.

Approve

Architecture direction

Business logic moves into @linky/core; the web app becomes an adapter and UI layer.

Approve

Migration order

Start with identity, contacts, and messaging protocol before payment execution.

Approve

Definition of done

Every chunk ships with tests, web-app integration, and removal of replaced duplicate logic.

Recommended first milestone

Complete the identity and owner-lane extraction, wire it through the current web app, and use that as the repeatable template for all later chunks.
